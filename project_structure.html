Codebase Structure & Dependencies

Overview

This project is an air quality monitoring, analysis, and reporting system. It has evolved over time, resulting in two distinct "paths" of execution: a modern path (active) and a legacy path (likely purely for ad-hoc or historical use).

Key Components

1. Active System (Modern Path)

The active system is modular and relies on air_module.py.

  • main_import_envi_data.py (Data Fetcher)
    • Role: The "Ingestor". It connects to the Ministry of Environmental Protection API.
    • Action: Runs continuously (loop), fetching data every 10 minutes.
    • Output: Appends raw data to output.csv.
    • Dependency: Imports air_module.py for data processing helper classes.
  • main_air_monitor.py (Scheduler & Reporter)
    • Role: The "Coordinator". It orchestrates analysis tasks.
    • Action: Uses the schedule library to run periodic analysis (half-hourly, 8-hour, daily).
    • Output: Generates Excel logs (log_params.xlsx) and Word reports with plots.
    • Dependency: Imports air_module.py for the heavy lifting (logic).
  • air_module.py (The Core Library)
    • Role: Shared library containing the logic.
    • Components:
      • data_importer: API connection and data fetching.
      • data_processor: JSON parsing and data cleaning.
      • data_analyzer: Statistical analysis and threshold checking.
      • data_plotter: Graphing and map generation.

2. Legacy System (Historical Path)

These files appear to be older versions or ad-hoc scripts. They do not use air_module.py but instead rely on Analize.py.

  • AnalizeShell_01.py: An older script for generating reports, similar to main_air_monitor.py but less modular.
  • Analize.py: An older version of the library. It contains the AnalizeAirData class, which has similar methods to air_module's data_analyzer but is less organized.
  • AnalizeAir.py: An even older, more procedural script for analysis.
